# 03. Development, Test & Ops – Dots and Boxes Game System

## 1. 개발 프로세스 (SDLC)

### 1.1 방법론
- Agile/Scrum 기반 2주 스프린트.
- 주요 산출물: 스프린트 목표, 백로그, 번다운 차트, 회고.

### 1.2 브랜치 전략
- main: 안정된 릴리즈 브랜치.
- develop: 통합 개발 브랜치(선택).
- feature/*: 기능 단위 작업.
- hotfix/*: 긴급 수정.

### 1.3 Antigravity + 바이브 코딩 가이드
- 작업 단위 명확히 정의 후 에디터/LLM와 상호작용.
- 자동 생성 코드도 PR 리뷰 필수.

## 2. 품질 & 테스트 전략

### 2.1 테스트 레벨
- Unit Test: 룰 엔진, 체인 탐지, AI 결정 로직.
- Integration Test: 클라이언트–서버 통신, 세션 라이프사이클.
- E2E/UI Test: 실제 플레이 흐름(튜토리얼 → 게임 → 결과).

### 2.2 AI/게임 특화 테스트
- 봇 난이도 검증: 각 레벨별 승률 범위 목표.
- 룰 위반 방지: 불법 수가 허용되지 않음을 자동화 테스트로 보장.
- 튜토리얼 플로우: 단계별 진입·이탈·재시작 시나리오.

### 2.3 품질 기준 (Quality Gate)
- 주요 기능 커버리지 목표.
- 크리티컬 버그 0, 룰 엔진 관련 버그 0 상태에서만 릴리즈.

## 3. Observability & SLO

### 3.1 SLI/SLO 정의
- 로그인 성공률, 게임 세션 생성 성공률.
- p95 응답 시간(세션 생성, move 요청).
- 튜토리얼 오류율.

### 3.2 모니터링
- 대시보드: 동시 접속, 에러율, 지연, 매치 대기 시간.
- 알림: SLO 위반 임계치에 도달 시 온콜 담당자에게 알람.

## 4. 배포 & 릴리즈 관리

### 4.1 환경 구분
- dev → stage → prod.
- 각 환경별 설정(엔드포인트, 키, DB) 분리.

### 4.2 배포 전략
- 소규모 트래픽: 롤링 배포.
- 점차 확장 시: 블루/그린 또는 카나리 적용.

### 4.3 릴리즈 프로세스
- 기능 완료 → 코드 리뷰 → 테스트 통과 → stage 배포 → QA 승인 → prod 배포.
- 릴리즈 노트 작성(기능, 버그 수정, 알려진 이슈).

## 5. 운영·Incident·CS

### 5.1 운영 Runbook
- 게임 서버 장애, DB 이슈, 높은 오류율 등 주요 시나리오별 대응 절차.
- 롤백 절차 및 기준 정의.

### 5.2 Incident Management
- 장애 발생 시 티켓 생성 → 등급 분류 → 대응 팀 할당.
- Incident 종료 후 Postmortem 작성(원인, 영향, 재발 방지 액션).

### 5.3 고객 지원 (CS)
- 인앱 문의 채널(폼/이메일 연동).
- FAQ/트러블슈팅 문서 운영.
- SLA(target): 예를 들어 영업일 24시간 이내 1차 응답.

## 6. 분석·실험 구조

### 6.1 이벤트 & 메트릭 수집
- 세션 시작/종료, 튜토리얼 단계, Move, 승패, 난이도, 체인 실수 패턴.

### 6.2 대시보드 & 리포트
- 리텐션, 세션 길이, 승률 편차, 튜토리얼 이탈 구간.

### 6.3 A/B 테스트 프레임
- 난이도/튜토리얼 변형 실험.
- 실험 결과를 기반으로 PRD/GDD 업데이트.
